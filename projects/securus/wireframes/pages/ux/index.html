Guidelines and Business Value

Navigation - Global

  • Global navigation will always provide the context of where a user is in the system.
  • The active class must be present for every global nav item.
  • Page Headers should always match the global nav area the page falls under.
Business value of global navigation

Provides context to the user; Increases efficiency of users; minimizes clicks to perform a goal;

Sub-Navigation

Active class must be present for sub global nav item.

Business value of sub navigation

Provides context for where users are in the system; provides navigation to specific pages; helps to more efficiently navigate; reduces call center volume

Tabs

  • Tabs should always contain a label.
  • If the tab contains multiple items, display the count in a badge next to the tab label.
  • An action taken in a primary table should activate the corresponding tab.
  • Active tab should have an Active state that matches the body of the panel.
Business value of tabs

Allows for aggregation of related information

Breadcrumbs

  1. Admin
  2. Facility and Sites
  3. CI Eden
  • Breadcrumbs will display the hierarchy from the parent page.
  • Parent = Global Nav; Global Nav > Level 1 > Level 2 > Current page
  • Display separator between each level in the navigation hierarchy.
Business value of breadcrumbs

Informs the user of their “whereabouts” while using the application

Search

Search

When typing Search Criteria and executing the search by pressing the button or hitting the ‘Enter’ key, a search is triggered.

  1. Upon entering search criteria and clicking Search, the system will narrow the returned results to only those where the number or name match part or all of the entered search criteria
    • Search will be contextual to the selected context of Facility and/or Site
    • Changing Facility/Site context will update the search results to show only those related to the new Facility and/or Site
  2. If no results are found, the system will display a message to the user that states:
    • "Your search did not match any records. Suggestions:
      • Make sure all words are spelled correctly.
      • Try different keywords.
      • Try a more general search."
  3. If results are found, the system will display the matches and list the number of results returned
Business value of search

Allows users to quickly locate and access information they are looking for

Tables

Show collections of items with multiple attributes in a data table to make the data easy to scan and consume.

  • Give context - Give the table a descriptive title, and also a brief caption if needed. Show the title at the top of the table or in the table header, and the caption below the table.
  • Compress data - Show only the information that users really need to see, but allow users to dig deeper.
  • Keep visual elements to a minimum - Remove all interface elements that are not necessary. Less visual noise means less distraction and makes it easier to concentrate on the table data. Focus should be on the information itself, not the design of the table. Don’t remove visual elements that are useful. Keep the elements that support structure, organization, and readability of the table, but use the minimum amount of visual weight needed
  • Emphasize the most important information - Use different styling for information that is always important.
    Note: If you emphasize too much information, nothing will stand out.
  • Allow sorting and filtering
Business value of tables

A well-designed table provides structure and makes it easy to scan information and find items from a large set of data.

Primary Buttons

  • Buttons should be readily identifiable as buttons.
  • Make sure users can find them easily.
  • The most important call to action should be the most prominent button.
  • Be sure the label for the button clearly indicates what the button is for.
  • Make destructive buttons the least prominent.
Business value of buttons

The button is the most common element of interaction making it easy for users to identify it as a call to action.

Notifications

Wireframe: #/pages/ux/system-feedback.html

The system will provide feedback for the following kinds of interactions:

  • system acting on a user interaction (e.g. user pressed the Save button).
  • system level actions that could affect the user (e.g. front end is unable to save a form because it can't communicate with the back end).

There will be four levels of feedback:

  1. Info: if the system wants to notify the user of something (e.g. "Password change might take 30min to propagate").
    Password changes can take up to 30 minutes to take effect.
    • Info notifications should not go away until the user presses the close button.
  2. Success: If the system successfully completed the user action (e.g. "New user created" or "Changes to inmate settings saved").
    Changes saved.
    • Success notifications will disappear automatically after 5 seconds.
    • Success notifications must appear above all other notification types.
  3. Alert: If a user performed an invalid action (e.g. user left a required field empty). Alert notifications should not go away until the user presses the close button.
    Please correct the errors in the form before saving.
  4. System: The front-end could not complete the action (e.g. front end couldn't save because it can't communicate with backend).
    We were unable to save the changes. Please try again later.
    • System notifications should not go away until the user presses the close button.
Multiple Notifications

If there are multiple notifications from different systems and the user has not closed an earlier notification, newer notifications should be stacked on top of older notifications. Closing the newest notification will reveal the next oldest one behind it, and so on.

Multiple Notifications, Same Level

If a user interaction (e.g. User presses Save) results in multiple notifications of the same level (alert/success/system) and same message e.g. "CDR Recording could not be downloaded", group them together as one notification with one message and a counter indicating the number of times the error appeared: e.g. "CDR Recording could not be downloaded (3)"

CDR could not be downloaded 3
Multiple Notifications, Different Levels
  1. If a single user interaction (e.g. User presses Save)
  2. Results in multiple notifications of different levels (e.g. Alert + Success, or Success + System)
  3. Then display a single notification with a message based on the following rule:
    • If (Alert + Success), display Alert level notification.
    • If (Success + System), display System level notification.
    • If (Alert + System), display System level notification.

e.g. If user hit the Save button, and part of the form could not be saved (because one part saves to Mongo, and another to AD), then:

  1. Only display the System level notification with a message "Unable to save some of your changes. Those fields are highlighted below."
  2. And highlight the fields that could not be saved with a validation message "Changes not saved" (see Form Validation).

If an error occurs because of user action (ie Save with required fields missing) and the user corrects the error in the page, the notification will be dismissed immediately.

Business value of notifications

Provides feedback to the user; draws attention to actions needed to complete a task; increases user memorability; enables users to operate more efficiently in the system

Form Design

Forms must always contain a label.

If a field is required, the field label must have a required class and follow the guidelines of form validation.

  • Email – Type=email; must contain validation for Email
  • Time – must contain a time picker; must contain validation for Time
  • Date – must contain a date picker; must contain validation for Date
  • Number – Type=number; must contain validation for Number; must contain validation for Min/Max values
  • Date/Time – must contain date/time picker; always displays icon to the right that launches control for selecting date/time

If we know in advance that a field may only contain a MAX number of characters, maxlength = “nnn” where nnn=max allowed characters.

Tab order will always be from top to bottom, left to right.

Business value of form design

Consistent behavior and appearance will allow users to more quickly become familiar with the look/feel/function of the application; Validation will ensure that user errors are caught on entry before they can reach the back end

Validation

Field is required

A field is in an invalid state if:

  1. A required field is left empty.
  2. If the value entered in a field does not match what is expected in that field (e.g. letters entered in a numeric field, number outside the allowed range.)
  • Saving a form
    • When I try to save a form, if there are fields in an invalid state, then the system should style the field using an invalid field class, and display a warning icon and an error message.
  • Tabbing off a required field
    • When a user presses the Tab key to jump to the next field in the form, and if the user left the field blank, and if the field is required, then the system should style the field using an invalid field class, and display a warning icon and an error message.
Business value of form validation

Reduces errors; reduces call center volume

Show/Hide

Show and Hide behaviors are used when content may or may not apply to the current screen based on other pieces of information selected or displayed. Show/Hide should be used when:

  • There are not large amounts of additional info/elements to reveal
  • Most users won’t need all the elements all the time
  • Elements are dependent or related to the value/content of primary elements on the page
Business value of show/hide

Using show/hide behavior to contextually display content and page elements helps keep the system simple and intuitive by only displaying the most essential and relevant content up front to the user. It also helps to logically group page elements that are related. Saves time and helps users avoid mistakes because there is less to scan and learn.

Checkboxes/Radio Buttons

Radio buttons are used when there is a list of two or more options that are mutually exclusive and the user must select exactly one choice. In other words, clicking a non-selected radio button will deselect whatever other button was previously selected in the list.

Checkboxes are used when there are lists of options and the user may select any number of choices, including zero, one, or several.

A stand-alone checkbox is used for a single option that the user can turn on or off.

Business value of checkboxes & radio buttons

Checkboxes and radio buttons visually present a range of choices as groups that are clearly separated from other groups on the same page. Radio buttons are preferable to dropdown menus because the options are always visible thus requiring a lower cognitive load.

Dropdowns

  • Use when there are more than two main sections of a site.
  • Use when there less than ten sections depending on the length of each section name.
Business value of dropdowns

While their use does not increase usability, they are a viable alternative when space is an issue.

Date/ Date Range Picker

Use when the user wants to easily choose a date or date range in order to submit, track, sort, or filter data.

Business value of date range pickers

The date picker is a familiar interface that is commonly understood among users. It helps the user easily choose a date or date range for use in submitting information or filtering data.

Time Picker

The time picker provides a familiar graphical interface helps users to easily apply or submit a time to events, tasks, appointments as well as items with chronological order.

Business value of time pickers

Smart defaults on time picker drop downs or entry fields guarantee data and spelling accuracy on user entry and minimize error.

Icons

Icons must first and foremost communicate meaning. A user’s understanding of an icon is based on previous experience.

To help compensate for icon ambiguity, a text label must be present alongside an icon to clarify its meaning in that particular context. Icon labels should be visible at all times, without any interaction from the user. For navigation icons, labels are crucial.

Business value of icons

Icons are easy to recognize, compact and visually pleasing for users.
