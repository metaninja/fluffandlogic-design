Guiding Principles

Overview

As a team we want to develop a consistent user interface when developing the EZLynx app. A consistent user interface is not only good for the user, it also helps us get features to our customers faster.

Good For The User

For the user, even if an experience is less than ideal, once they learn how to use it, they can reuse that knowledge elsewhere in the app. A consistent experience gives the appearance of a polished robust app, which helps with branding and marketing.

Good For Development

For development, consistency saves time by allowing us to reuse existing styles and features, which means more time working on features, and less time tweaking structural and presentation code, which makes development more manageable over the long term.

Tl:dr; Spend less time writing HTML code, more time working on the functionality.

Why Guiding Principles?

Guiding principles help unite Product, Development with a common, shared goal that we can all aspire to. They help us figure out what to prioritize when designing the UI for the app.

  1. Start With Workflows and Mental Models

    At EZLynx,UXPin is our preferred tool for ideating or designing UIs. UXPin makes it very easy to jump right in and come up with a solution for a customer feature or problem. Too easy, in fact.

    The problem is, if we don't spend some time up front thinking about what we're going to build, we can easily end up solving the wrong problem, or solving the problem wrong, which increases support calls and might mean additional development time to fix the problem we initially set out to solve.

    Before you jump into UXPin, spend some time sketching out the user/process/work flow on a whiteboard. Where will the user start to get to your feature? What steps will they follow to get there? What is the first thing they need to do?

  2. Test With Users Before Development

    Development is expensive. Once a team has started developing a feature, it's highly unlikely we'll be able to make major changes.

    With that in mind, once your mockups and prototypes look good enough for development, give some users a call, and let them test drive your features in UXPin or via our prototype, and give you feedback. You only need 3-5 users to solve nearly 85% of the user issues that crop up with most new features.

  3. Favor Reuse And Consistency Over Innovation

    When creating a new feature, try to use the existing building blocks (colors, styles, and components) as much as possible. Only if the existing standards aren't adequate, or if it will result in a worse user experience, should you think about creating innovative new UIs that use controls or colors not already available in the app.

    For instance, do we already have a file uploader in the system? Yes, we do. In fact we have FIVE. When working with development, look for ways to reuse features we've built or used elsewhere in the system.

  4. Strive For Clarity Over Brevity

    Whenever possible, be very clear about what action you're about to perform.
    E.g. Labeling a button "Clear Login" instead of "Login".

    Note that the button should be concise and not verbose.
    e.g. "Clear Login" instead of "Please Click Here To Clear Your Login".

  5. Use Icons Instead of Images

    The EZLynx app ships with a large set of icons.
    With a few rare exceptions (e.g. the icons in the Applicant Overview sidebar) in the app, use icons instead of bitmap graphics like PNGs.
